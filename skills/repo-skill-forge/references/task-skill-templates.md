# Task Skill Templates

Authoritative spec for each skill generated by `repo-skill-forge`.
For each skill type: content sources, section structure, quality checks.

**`{repo-slug}` in every template below** = the repo name lowercased, with every
run of non-`[a-z0-9-]` chars replaced by a single hyphen, trimmed of leading/
trailing hyphens (e.g. `acme_app` → `acme-app`). Every generated skill's
folder AND its frontmatter `name` MUST equal `{repo-slug}-<skill>` — agentskills
names are lowercase letters/numbers/hyphens only and MUST equal the folder. A
colon-namespaced name (`wicked-understanding:fix-bug`) is invalid and won't install.

The generated skills progressively load deeper context from
`../{repo-slug}-wiki/refs/` — the wiki backing store built by `repo-wiki-planner`,
which installs as a sibling folder named `{repo-slug}-wiki`.

---

## Read first: these examples are ILLUSTRATIVE, not prescriptive

The code blocks below skew toward one stack (TypeScript / REST / handler →
service → repository with a DI container and an ORM). That is **one** shape, not
the default. Before writing any skill, read `survey.md` + the four lenses and
pin down the repo's actual paradigm:

- **Language** — TS, Go, Rust, Python, Java, …
- **Style** — OO vs functional-core vs procedural
- **Shape** — web service vs CLI vs library vs batch/pipeline
- **Persistence** — has-DB (+ORM/migrations) vs no-DB / in-memory / file-backed
- **Layout** — layered (transport/app/domain/infra) vs flat / module-per-feature

Map every section to **that** reality. If the repo has no controllers, no DI
container, no ORM, and no migrations, **do not invent them** — describe the real
structure instead:

| Stack | "Add a unit of work" reality |
|---|---|
| Go (CLI/server) | new `<name>_tool.go` (or `<name>.go`) + register in `server.go` / `main.go`; build with `go build ./...` |
| Rust | new module + `impl Trait for T`, wire in `mod.rs` / `lib.rs`; build/test with `cargo` |
| Python (functional core) | new pure module, wired explicitly at the composition root; no DI annotations |
| TS/NestJS (the example below) | controller → service → repo + provider registration |

A skill that cites migrations, a DI container, or controllers in a repo that has
none is **filler, not a default** — it fails the "no name-swappable sentences"
bar and makes the skill actively wrong. When the survey contradicts a template
section, follow the survey and drop or rewrite the section.

## verify

**When it fires**: agent (or a person) is confirming a change actually works,
deciding whether work is "done", reviewing/accepting a change, or picking up an
unfamiliar capability and needing to test it.

**Not** `write-tests`. That skill is *how to author a test* (framework, fixtures,
where files go). This skill is *what correct behavior looks like and how to prove
it* — the behavioral spec plus the gate sequence. They cross-reference; they don't
overlap.

**Always-in-context goal**: a tester who has **never read the implementation** can
name what each capability should observably do, exercise it by hand, run the right
gates in order, and apply a concrete definition of done. This is the skill that
turns "the code" into "behavior anyone can verify."

**Boundary — this is knowledge, not a runner.** It describes what to check and how
to exercise it so *any* consumer (a person, an agent, or a delivery orchestrator)
can verify. It must **not** name or depend on an external acceptance/orchestration
tool — the repo's own test framework (the real `pytest`/`go test`/`cargo test`) is
fine; a sibling product is not. That keeps the generated package self-contained.

## Gate sequence — run before calling a change done

In order; each is a gate. **Include only the gates this repo actually has —
derive commands from ops.md + technical.md. Do not invent a typecheck step for an
untyped repo or an e2e suite that doesn't exist; drop the row instead. Gates may
be conditional — some repos run one only when a certain area changed (coverage if
code changed; a browser/e2e suite only if the UI or a route changed). If so, note
the trigger in the row; don't imply every gate runs on every change. If no
artifact states a gate's trigger condition, derive it from what the gate
exercises (a browser/e2e gate → UI/route changes; a coverage gate → code changes)
and mark it *inferred* — so two operators reach the same answer instead of one
marking it "always".**

## Shared: description formula

Every generated skill description must follow this formula, emitted as a YAML
**folded block scalar** (`description: >`):

```
description: >
  {Task} workflow for {repo-name} — a {type} built with {stack}.
  Use when {specific trigger condition}.
  Covers: {3 specific things this skill knows about this repo}.
  Load {before/when} {specific agent action} in {repo-name}.
```

The generated `description:` MUST be a folded block scalar (`>`), because the
formula contains `: ` (`Covers:`, `Use when …:`) which breaks a plain YAML
scalar — `npx skills` throws "Nested mappings are not allowed in compact
mappings" and SILENTLY DROPS the skill (exit 0, skill never installs). The `>`
on its own line, with the text indented beneath it, makes every colon in the
body literal and safe.

The third sentence MUST name specific, repo-specific facts — not generic
descriptions. Pull the facts from the repo's actual stack/shape, e.g.:
- TS/REST: "the Zod-validated request pipeline, hexagonal layer triage, and pnpm test commands"
- Go CLI/server: "the tool-registration step in `server.go`, `error`-return propagation, and `go test ./...`"
- Rust library: "the trait-impl wiring in `lib.rs`, `Result`/`?` error flow, and `cargo test`"

Bad (name-swappable, fits any repo): "how to fix bugs and write tests".
