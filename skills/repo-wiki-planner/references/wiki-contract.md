# Wiki Output Contract

Every article generated by `repo-wiki-planner` MUST follow this contract.
Read this before writing any article. The lint self-check at the end is
mandatory before saving.

Source of truth: the uploaded `wiki-system/SKILL.md` and `wiki-system/output-expectations.md`.
This file is a working condensation; defer to those files if there's a conflict.

---

## Frontmatter (required at top of every article)

```yaml
---
title: <Title Case, 3-7 words, no trailing period>
slug: <purpose-prefixed slug — must match the slug arg to wiki_article_publish>
canonical_for:
  - <REPO>-<PURPOSE>-<NOUN>          # IDs this page OWNS. One per article.
references:
  - <REPO>-OVERVIEW                  # IDs/pages this article cites — populate from See Also
audience: maintainer                 # 'maintainer' or 'user' or ['maintainer','user']
tier: 1                              # 1 for AI-generated with analysis grounding
generators: []                       # empty — not generated by a deterministic extractor
source_hash: ""                      # not required for AI-generated pages
---
```

### Canonical ID format

`{REPO}-{PURPOSE}-{NOUN}` — all caps, hyphens between segments.

| Purpose segment | Use when |
|---|---|
| `OVERVIEW` | product overview page |
| `ARCH` | architecture overview |
| `API` | API reference |
| `CAP-{FEATURE}` | capability article |
| `CONCEPT-{TERM}` | concept explanation |
| `ONBOARDING-{AUDIENCE}` | onboarding article |
| `PATTERN-{PATTERN}` | design pattern |
| `RB-{OPERATION}` | runbook |
| `AGENTS` | agent roster |

Examples: `PAYMENTS-OVERVIEW`, `PAYMENTS-CAP-CHECKOUT`, `PAYMENTS-API`, `PAYMENTS-RB-MIGRATE`.

### Required vs optional frontmatter fields

- **Required**: `title`, `slug`, `tier`
- **Required if owns canonical IDs**: `canonical_for`
- **Required if body cites other canonical IDs**: `references`
- **Required always**: `audience`, `generators: []`

## H2 anchor pattern (strict order, required)

Every article uses these H2 anchors. Omit empty sections — don't emit a stub.

```markdown
# {Title}

(No preamble. No "This page describes…" or "In this article…")

## Purpose
What this page owns. 1–3 sentences. Required if canonical_for is set.

## Contract
The interface, structure, or content this page documents.
Use H3 sub-sections within Contract for complex articles.
Each entry ends with a [src: file:{path}] citation.

## Invariants
Properties that MUST hold. Each is one sentence ending in a period.
Cite the file where the invariant is enforced: [src: file:{path}:{line}]

## Gotchas
Known traps, edge cases, surprising behavior. 2–4 sentences per gotcha.
Each gotcha cites its source.

## See also
Cross-references. Format:
- [{CANONICAL-ID}](#{slug}) — one-sentence why
```

**Forbidden H2 headings**: `## Overview`, `## Architecture`, `## Components`,
`## Top-level Modules`, `## Background`, `## Introduction`. These rephrase
the contract instead of contributing to it. Use H3s inside `## Contract` instead.

## Citation format

Only one citation namespace is available for repo-wiki-planner (no DB UUIDs):

| Type | Format | Use when |
|---|---|---|
| File | `[src: file:{repo-relative-path}:{line-or-range}]` | Citing a specific source file |
| File (directory) | `[src: file:{repo-relative-dir}/]` | When exact file is known but not line |
| Canonical-ID | `[src: {REPO}-{PURPOSE}-{NOUN}]` | Citing another page in this wiki batch |

`[src: ...]` appears at the end of the sentence it grounds, before the period.
Multiple citations: comma-separate inside one bracket.

**Do NOT use** `[src: memory:<uuid>]`, `[src: symbol:<uuid>]`, or
`[src: document:<uuid>]` — these are for live tool sessions with DB access,
not file-grounded analysis.

**Good**: "The Repository pattern is used for all DB access (`[src: file:src/repos/]`)."
**Bad**: "The repository layer handles persistence." (no citation)
**Bad**: "See the relevant code." (no citation)

## Closer (every article — last block)

```markdown
---

## Evidence summary
- Analyzed: {list the specific analysis artifacts used, e.g. "architecture.md, domain.md"}
- Key references: {3–5 most load-bearing sources — specific file paths from the analysis}
- Coverage gaps: {what you couldn't verify — e.g. "exact line numbers not confirmed; exact migration command not verified"}

## Open questions
- {specific deferred question — e.g. "Is the `{Foo}` entity actually persisted or just in-memory?"}
- {another question if applicable}

## Confidence
HIGH | MEDIUM | LOW — {one-sentence why, tied to evidence quality}
```

Rules:
- `(none)` is acceptable for Open Questions and Evidence gaps if genuinely empty.
  Write `(none)` explicitly — omitting looks like the writer forgot.
- `LOW` confidence is acceptable here. It forces the gap into the open.
- Confidence = HIGH when all required sections are grounded with real file paths.
  MEDIUM when 1–2 sections are inferred. LOW when a primary artifact was unavailable.

## Lint self-check (run before every publish)

Check each of the 5 rules. Record results and fix any failures before saving.

| Rule | Level | Check |
|---|---|---|
| `duplicate_canonical_for` | error | Does any `canonical_for` ID appear on another article in this batch? |
| `broken_reference` | error | Does every `references` entry point to a canonical-ID declared in this batch or the wiki? |
| `long_page_low_refs` | warn | If page > 80 lines AND `references` < 3 entries AND `canonical_for` < 2 IDs → add references or trim |
| `missing_canonical_purpose` | error | If `canonical_for` is set → does the article have a `## Purpose` H2? |
| `unresolvable_citation` | n/a | No UUID citations in repo-wiki-planner output — this rule doesn't fire |

Self-check format to include in the generation log (not in the article):
```
Lint check: {article-slug}
  duplicate_canonical_for: PASS
  broken_reference: PASS (references: {list})
  long_page_low_refs: PASS (lines: {N}, refs: {N})
  missing_canonical_purpose: PASS
  unresolvable_citation: N/A
```

## Forbidden phrasing (universal)

- "This page describes…" / "In this article we'll cover…" — frontmatter establishes scope
- Future tense for current behavior ("the system will handle…" → "the system handles…")
- "It seems that…" / "Probably…" / "I think…" — use the Confidence closer instead
- "See the relevant code" without a `[src: ...]`
- "comprehensive", "robust", "seamless", "cutting-edge", "sophisticated", "leverage" (as a verb)
- "AI-powered" — name the model or mechanism
- "Future Enhancements" / "Roadmap" / "Coming soon" — articles describe what exists
- Any sentence that reads unchanged if you swap the project's name — it's filler

---

## Writing style

- Imperative when instructing: "Call `POST /api/v1/...`" not "You can call..."
- Declarative when describing: "The runtime strips tools not in the allow-list"
- Specific nouns: "`agent_sessions` table" not "the database"
- Quantify: "18 endpoints" not "many endpoints"
- Kill passive voice when subject is known
- One idea per paragraph

---

## Audience rules

**`audience: maintainer`**
- Has file access, can run grep
- Cite internal file paths: `[src: file:src/services/user.service.ts:42]`
- Include implementation detail: class names, method names, table names

**`audience: user`**
- No source access; cannot open files
- Cite only observable surfaces: API endpoints, UI paths, config keys
- Never include `[src: file:...]` paths in user-audience articles
- Every claim grounded in what the user sees, calls, or configures

**`audience: both` / `['maintainer','user']`**
- One article that serves both; use the maintainer citation level
- Mark sections that are platform-user-relevant with "(for users:)"
- Or split into two articles if the content genuinely diverges
